Projektanlass & Ausgangslage

Anlass: Nuxt 4 veröffentlicht; Support für Nuxt 3 endet zum 31.07.2026

  • Migrationen rein manuell oder mit Coding-Agenten — aber ohne einheitliches Prompting
  • Hoher Zeitaufwand bei wiederkehrenden Aufgaben — Leitfäden und Modul-Docs jedes Mal neu zusammentragen
  • Unterschiedliche Ergebnisse pro Migration — im Code Review schwer nachvollziehbar
Kernfrage: Wie lassen sich wiederkehrende Migrationsaufgaben optimieren, um Zeitaufwand und Doppelarbeit zu reduzieren?

Vorgehensmodell & Projektplanung

Erweitertes Wasserfallmodell — klare Phasen, iterative Umsetzung an echten Test-Repositories

  • Klassische Phasen: Analyse → Entwurf → Implementierung → Test → Dokumentation
  • Implementierung iterativ — Erkenntnisse aus Test-Repos fließen zurück in Code und Planung
  • Pflichtenheft und IHK-Dokumentation erfordern Planbarkeit; technische Details mussten an realen Nuxt-Projekten validiert werden
  • Geplanter Projektumfang von 80 Stunden wurde eingehalten

Architektur & Technologie

Modularer Schichtenaufbau — offizielle Nuxt-Tools bleiben die technische Basis

  • Stack: TypeScript, Node.js, @clack/prompts — stateless, keine Datenbank
  • Fünf Bausteine: CLI · Projektanalyse · Modul-Workflow · Artefakte · Agenten
  • Drei Workflows: Kernframework · offizielle Module · kombiniert
  • Nuxt-Kommandos (nuxi upgrade, Typecheck, Build) werden orchestriert, nicht ersetzt

Implementierung: CLI & Projektanalyse

Umsetzung: Interaktive CLI mit kontextsensitiver Projektanalyse

  1. Projektpfad wählen und Zielprojekt validieren
  2. Vorprüfung: package.json, Nuxt-Abhängigkeit, Package-Manager
  3. Workflow auswählen (Kern / Module / kombiniert)
  4. Kritische Schritte nur nach expliziter Bestätigung
  5. Upgrade-Schritte ausführen, protokollieren und validieren
  • Unterstützte Package-Manager: npm, pnpm, yarn, bun
  • Automatisches Setzen von future.compatibilityVersion = 4 wo sinnvoll

Modul-Workflow & Agenten

Strukturierter Modul-Workflow — optional Agenten für Restarbeit

  • Scan aller Nuxt-Module → Modul-Inventar als Artefakt für Code Review
  • Migrationshinweise aus offiziellen Guides aggregiert — kein manuelles Zusammentragen
  • Modulspezifische Änderungen nachvollziehbar — keine blinde Massenautomatisierung
  • Optionale Agenten-CLIs mit begrenztem Scope — Entwickler behält die Kontrolle

Validierung, Artefakte & Qualitätssicherung

Nachvollziehbarkeit als Qualitätsziel — jeder Lauf hinterlässt überprüfbare Spuren

Validierung

  • Typecheck und Build direkt im Workflow integriert
  • Fehler über Exitcodes und Logs dokumentiert

Artefakte

  • Logs, Migrationsberichte, Modul-Inventare
  • Agenten-Instruktionen für Folgearbeiten

Qualitätssicherung

  • Nachweis ausgeführter Schritte im Code Review
  • Testläufe an mehreren internen Nuxt-3-Repositories
  • Nicht nur Happy Path — unterschiedliche Modul-Komplexität

Fazit, Lessons Learned & Ausblick

Standardisierung ohne Kontrollverlust — erweiterbar für künftige Nuxt-Versionen

Ergebnis

  • Geführter Migrationsprozess mit voller Entwicklerkontrolle
  • Weniger Rechercheaufwand, einheitlicher Ablauf im Team

Lessons Learned

  • Automatisierung ersetzt keine fachliche Modulprüfung
  • Frühe Tests an echten Repositories sparen späte Korrekturen

Ausblick

  • Weitere Modulregeln und Validierungsschritte
  • Übertragbar auf spätere Nuxt-Versionen
  • Bereitstellung als npm-Paket im Unternehmen
